## 1. The Linux Input and Display Stack: Key Components

### 1.1 X11 and Wayland

- **X11 (X Window System):** The traditional and widely used display server protocol in Linux. It handles:
  - Display rendering
  - Input device management (keyboard, mouse)
  - Communication between graphical applications and the hardware
- **Wayland:** A newer display server protocol designed to replace X11, aiming for improved performance and security. Wayland also manages display and input devices but differs fundamentally in architecture and support.
  
**Key Point:** Many keyboard remapping tools (e.g., `xmodmap`, `setxkbmap`) are built specifically for X11 and do **not** work under Wayland.

### 1.2 GNOME Desktop Environment

- GNOME runs on top of either X11 or Wayland.
- It provides graphical UI elements, session management, and input method integration.
- GNOME settings and utilities may override or interact with underlying keyboard configurations.

## 2. Keyboard Mapping Mechanisms and Tools

### 2.1 Keyboard Input Flow (Simplified)

Hardware keyboard → Kernel input subsystem (evdev/libinput) → X11/Wayland → Desktop environment (GNOME) → Applications

### 2.2 Key Mapping Files and Tools

- **`/usr/share/X11/xkb/symbols/us`:**  
  A system-level X11 keyboard layout file defining key mappings and behavior for the US keyboard layout.  
  *Modifying here changes the key behavior for all X11 sessions using that layout.*

- **`~/.Xmodmap`:**  
  User-level configuration file for `xmodmap`, a tool to dynamically remap keys in X11 sessions.  
  *Non-persistent by default; must be loaded on session start.*

- **`~/.xprofile`:**  
  A user script executed at login to run commands like `xmodmap ~/.Xmodmap` for applying key mappings automatically.

- **`~/.config/autostart/*.desktop`:**  
  Desktop Entry files to run commands automatically on graphical session start. For example,  
  `setxkbmap us` can be called here to set a specific keyboard layout and re-load `/usr/share/X11/xkb/symbols/us`.

- **`setxkbmap`:**  
  Command-line tool to set the keyboard layout, options, and variants for X11. It directly applies layout changes for the session.

## 3. Why Did Your First Method (`xmodmap`) Fail to Persist?

- `xmodmap` applies changes dynamically **but does not alter the underlying system keyboard layout**.
- If not loaded properly on every session start (e.g., via `.xprofile` or autostart), changes revert.
- Modern desktop environments and input systems (especially with Wayland) may ignore or override `xmodmap` settings.
- System or session-level settings like GNOME keyboard options or low-level XKB layouts take precedence.

---

## 4. Why Does Editing `/usr/share/X11/xkb/symbols/us` Work?

- This is a **permanent system-level modification** of the X11 keyboard layout.
- All sessions using the `us` layout inherit this change, making the remap persistent and consistent.
- When `setxkbmap us` is run (e.g., from autostart), it reloads this modified layout.
- This method is more robust but requires administrative privileges and is less flexible for quick changes.

---

## 5. What Does `gsettings set org.gnome.desktop.input-sources xkb-options "['ctrl:nocaps']"` Do?

- This command uses **GNOME's settings daemon** to apply keyboard options within the GNOME desktop environment.
- `ctrl:nocaps` is a predefined option that remaps Caps Lock to Ctrl.
- GNOME applies this at the desktop session level, overriding X11 or Wayland defaults.
- This is why it's simpler and immediate for common remaps, without manual file edits or scripts.
- It leverages **higher-level, user-friendly configuration**, abstracting the underlying complexity.
